===========================================
MELIUS BACKEND - DOKUMENTACJA TECHNICZNA
===========================================

OPIS OGÓLNY:
------------
Melius to platforma do łączenia pacjentów z terapeutami online. Backend oparty jest na NestJS, TypeScript, PostgreSQL i Prisma ORM. 
Aplikacja obsługuje trzy role użytkowników: user (pacjent), doc (terapeuta), admin (administrator).

STACK TECHNOLOGICZNY:
---------------------
- Framework: NestJS
- Język: TypeScript
- Baza danych: PostgreSQL
- ORM: Prisma 7
- Autentykacja: JWT + cookies
- Płatności: Stripe, Przelewy24
- Email: Resend
- Storage: AWS S3 (awatary i dokumenty weryfikacyjne)
- Harmonogramowanie: @nestjs/schedule (cron jobs)

STRUKTURA BAZY DANYCH:
----------------------

ENUM TYPES:
- Role: user, doc, admin
- Sex: male, female, other
- MeetingStatus: pending, confirmed, cancelled_by_user, cancelled_by_doc, cancelled_by_system, completed
- PaymentStatus: pending, succeeded, failed, refunded
- PaymentProvider: stripe, p24
- Profession: psychologist, psychotherapist, sexologist
- Language: pl, en
- DocVerificationStatus: draft, submitted, approved, rejected

GŁÓWNE MODELE:

1. RefreshToken (Sesja odświeżania)
   - tokenHash, expiresAt, revokedAt, replacedByTokenId
   - userAgent, ip
   - Relacje: user, replacedBy, replaces

2. User (Użytkownik)
   - id, email, password, role
   - emailConfirmed (weryfikacja emaila)
   - language (preferowany język)
   - tokenActivatedAt (ostatnia aktywacja tokenu)
   - Relacje: avatar, userProfile, docProfile, adminProfile

3. UserProfile (Profil pacjenta)
   - firstName, lastName, published (pole legacy, nieużywane w logice)
   - consentTerms, consentAdult, consentHealthData (zgody z register-light)
   - Relacje: user, meetings, problems, ratings

4. DocProfile (Profil terapeuty)
   - firstName, lastName, profession
   - rate (średnia ocena), ratesLot (liczba ocen)
   - sessionPricePln (cena sesji w PLN)
   - docTermsAccepted (zgoda terapeuty na regulamin)
   - verificationStatus (draft, submitted, approved, rejected)
     * draft - profil w trakcie tworzenia
     * submitted - przesłany do weryfikacji
     * approved - zatwierdzony, widoczny publicznie
     * rejected - odrzucony przez admina
   - submittedAt, reviewedAt, rejectionReason
   - Relacje: user, meetings, specializations, availabilitySlots, ratings, docVerificationDocuments

5. DocVerificationDocument (Dokument weryfikacyjny)
   - fileKey, mimeType, sizeBytes, originalName, uploadedAt
   - Relacje: doc

6. AdminProfile (Profil administratora)
   - firstName, lastName
   - Relacje: user

7. Avatar (Awatar)
   - key, url
   - Relacje: user

8. Meeting (Spotkanie)
   - startTime, endTime, status
   - clientMessage (wiadomość od klienta)
   - Relacje: user, doc, slot, payments, ratings

9. AvailabilitySlot (Slot dostępności terapeuty)
   - startTime, endTime, booked
   - Relacje: doc, meetings

10. Payment (Płatność)
   - unitAmount, currency, status, provider
   - stripePaymentIntentId, stripeCheckoutId (Stripe)
   - p24SessionId, p24OrderId, p24Token (Przelewy24)
   - Relacje: meeting

11. Rating (Ocena)
   - rate (1-5), comment
   - Relacje: user, doc, meeting

12. Problem (Problem pacjenta)
   - problemKey (unikalny klucz)
   - Relacje: specializations, userProfiles, matches

13. Specialization (Specjalizacja terapeuty)
   - specializationKey (unikalny klucz)
   - Relacje: problems, docProfiles, matches

14. Match (Dopasowanie problem-specjalizacja)
    - problemId, specializationId

===========================================
MODUŁY I ENDPOINTY API
===========================================

1. AUTH MODULE (/auth)
----------------------
Obsługuje rejestrację, logowanie, wylogowanie i ustawianie hasła.

ENDPOINTY:
----------

POST /auth/login
- Body: { email, password }
- Zwraca: role i userId
- Logika: Weryfikuje email i hasło, sprawdza czy email jest potwierdzony
- Cookie: access_token (httpOnly), refresh_token (httpOnly, path=/auth/refresh),
  isAuth, access_exp, refresh_exp

POST /auth/register-light
- Body: { email, firstName?, lastName?, consentTerms, consentAdult, consentHealthData }
- Zwraca: role i userId
- Logika: "Lekka" rejestracja bez hasła - pozwala użytkownikowi rozpocząć proces
  bez pełnej rejestracji. Jeśli użytkownik już istnieje z niepotwierdzonym emailem,
  aktualizuje jego dane i loguje go.
- Email: wysyła link potwierdzający (ustaw hasło / potwierdź email)
- Wymagane zgody (wszystkie true): regulamin + polityka prywatności, pełnoletniość
  lub zgoda opiekuna, zgoda na przetwarzanie danych dotyczących zdrowia.
- Cookie: access_token (httpOnly), refresh_token (httpOnly, path=/auth/refresh),
  isAuth, access_exp, refresh_exp

POST /auth/set-password (wymaga JWT)
- Body: { password }
- Zwraca: role i userId + nowe cookies
- Logika: Ustawia hasło dla użytkownika z "lekkiej" rejestracji i potwierdza email

POST /auth/refresh
- Zwraca: { ok: true } + nowe cookies
- Logika: Rotuje refresh token i wydaje nowy access token
- Wymaga: refresh_token w cookie (path=/auth/refresh)

POST /auth/logout
- Zwraca: { ok: true }
- Logika: Czyści cookies i unieważnia refresh token (jeśli obecny)

AUTH SERVICE - METODY:
----------------------
- login(email, password): Weryfikuje dane i wydaje sesję (access + refresh)
- registerLight(email, firstName?, lastName?): Tworzy "lekkiego" użytkownika bez hasła
- setPassword(userId, password): Ustawia hasło i potwierdza email
- refreshSession(refreshToken): Rotacja refresh tokenu
- revokeRefreshToken(refreshToken): Unieważnia refresh token

2. USERS MODULE (/users)
-------------------------
Zarządzanie użytkownikami - CRUD operations, awatary.

ENDPOINTY:
----------

POST /users
- Body: CreateUserDto (email, password, role, language,
  consentTerms, consentAdult, consentHealthData)
- Zwraca: Utworzonego użytkownika (bez hasła)
- Logika: Tworzy nowego użytkownika z profilem odpowiednim dla roli
  - Dla roli user wymagane są zgody (wszystkie true)

GET /users/me (wymaga JWT)
- Zwraca: Dane zalogowanego użytkownika
- Guard: JwtAuthGuard

GET /users (wymaga JWT + admin role)
- Zwraca: Lista wszystkich użytkowników
- Guard: JwtAuthGuard, RolesGuard

GET /users/:id (wymaga JWT + admin role)
- Zwraca: Dane konkretnego użytkownika
- Guard: JwtAuthGuard, RolesGuard

PATCH /users/:id (wymaga JWT, tylko własne dane)
- Body: UpdateUserDto (email?, password?, language?)
- Zwraca: Zaktualizowanego użytkownika
- Logika: Użytkownik może aktualizować tylko swoje dane

DELETE /users/:id (wymaga JWT, tylko własne konto)
- Zwraca: Usunięty użytkownik
- Logika: Usuwa użytkownika i czyści cookie, kaskadowo usuwa powiązane dane

POST /users/avatar (wymaga JWT)
- Body: Multipart form-data z plikiem (limit 2MB)
- Zwraca: Avatar (key + url)
- Logika: Upload do S3, aktualizuje rekord Avatar w bazie

DELETE /users/avatar/:userId (wymaga JWT)
- Zwraca: Status usunięcia
- Logika: Usuwa plik z S3 i rekord z bazy (tylko własny userId)

USERS SERVICE - METODY:
------------------------
- create(dto): Tworzy użytkownika z odpowiednim profilem
- findAll(): Zwraca wszystkich użytkowników
- findOne(id): Zwraca użytkownika po ID
- update(id, dto): Aktualizuje dane użytkownika (email, hasło, język)
- remove(id): Usuwa użytkownika
- addAvatar(userId, file): Upload awatara do S3 (normalizacja do webp)
- deleteAvatar(userId): Usuwa awatar z S3 i bazy

3. PROFILES MODULE (/profiles)
-------------------------------
Zarządzanie profilami użytkowników, terapeutów i adminów.

ENDPOINTY:
----------

PUT /profiles/user (wymaga JWT + user role)
- Body: UpdateUserProfileDto (firstName, lastName, problems[])
- Zwraca: Zaktualizowany profil użytkownika
- Logika: Aktualizuje dane pacjenta i przypisuje problemy

PUT /profiles/doc (wymaga JWT + doc role)
- Body: UpdateDocProfileDto (firstName, lastName, profession, sessionPricePln, 
        specializations[], docTermsAccepted?)
- Zwraca: Zaktualizowany profil terapeuty
- Logika: Aktualizuje dane terapeuty, specjalizacje i cenę sesji
- Uwaga: Nie można zmienić verificationStatus tym endpointem

POST /profiles/doc/submit (wymaga JWT + doc role)
- Zwraca: Zaktualizowany profil terapeuty
- Logika: Przesyła profil terapeuty do weryfikacji (verificationStatus=submitted)
- Walidacja: Wymaga wypełnienia firstName, lastName, profession, sessionPricePln,
  specializations, docTermsAccepted=true oraz co najmniej 1 dokumentu weryfikacyjnego
- docTermsAccepted: akceptacja regulaminu dla terapeutów (wymagana przed submit)

POST /profiles/admin (wymaga JWT + admin role)
- Body: UpdateAdminProfileDto (firstName, lastName)
- Zwraca: Zaktualizowany profil admina

GET /profiles/user (wymaga JWT + user role)
- Zwraca: Profil zalogowanego pacjenta z problemami

GET /profiles/doc (wymaga JWT + doc role)
- Zwraca: Profil zalogowanego terapeuty ze specjalizacjami

POST /profiles/rate (wymaga JWT + user role)
- Body: CreateDocRateDto (docId, meetingId, rate, comment?)
- Zwraca: Zaktualizowany profil terapeuty
- Logika: Tworzy ocenę, aktualizuje średnią ocenę terapeuty

GET /profiles/doc/:docId/ratings
- Query: page=1, limit=10
- Zwraca: Lista ocen terapeuty z paginacją
- Logika: Pobiera oceny terapeuty posortowane od najnowszych

PROFILES SERVICE - METODY:
---------------------------
- updateUserProfile(userId, dto): Aktualizuje profil pacjenta
- updateDocProfile(userId, dto): Aktualizuje profil terapeuty (bez verificationStatus)
- submitDocProfile(userId): Przesyła profil terapeuty do weryfikacji
- updateAdminProfile(userId, dto): Aktualizuje profil admina
- getUserProfile(userId): Pobiera profil pacjenta
- getDocProfile(userId): Pobiera profil terapeuty
- getNewDocsProfiles(): Pobiera nowych terapeutów (tylko approved, używane wewnętrznie)
- getBestDocsProfiles(): Pobiera najlepiej ocenianych terapeutów (tylko approved, używane wewnętrznie)

RATING SERVICE - METODY:
-------------------------
- addDocRate(user, createDocRateDto): Tworzy ocenę i aktualizuje średnią
- getDocRatings(docId, page, limit): Pobiera oceny terapeuty z paginacją

4. MATCHES MODULE (/matches)
-----------------------------
System dopasowywania pacjentów do terapeutów na podstawie problemów i specjalizacji.

ENDPOINTY:
----------

GET /matches (wymaga JWT + user role)
- Zwraca: Lista dopasowanych terapeutów
- Logika: Znajduje terapeutów, których specjalizacje pasują do problemów użytkownika
- Filtr: verificationStatus=approved
- Sortowanie: Po rate (malejąco)
- Fallback: jeśli brak dopasowań, zwraca listę nowych terapeutów (internal)

GET /matches/problems
- Zwraca: Lista wszystkich problemów z bazy
- Publiczny endpoint

POST /matches/search-with-slots
- Body: SearchMatchesDto (problemId, from, to)
- Zwraca: Lista terapeutów z wolnymi slotami
- Logika: Znajduje terapeutów pasujących do problemów, którzy mają wolne sloty
  w podanym okresie
- Filtr: verificationStatus=approved

MATCHES SERVICE - METODY:
--------------------------
- getMatchedDocs(userId): Dopasowuje terapeutów do problemów użytkownika (tylko approved)
- getProblems(): Zwraca wszystkie problemy
- searchDocsWithSlots(dto): Wyszukuje terapeutów z wolnymi slotami (tylko approved)

5. AVAILABILITY MODULE (/availability)
---------------------------------------
Zarządzanie dostępnością terapeutów (sloty czasowe).

ENDPOINTY:
----------

POST /availability (wymaga JWT + doc role)
- Body: CreateAvailabilityDto (startTime, endTime)
- Zwraca: Utworzony slot
- Logika: Tworzy nowy slot dostępności dla terapeuty
- Walidacja: 
  * Wymaga verificationStatus=approved
  * Sprawdza czy slot nie koliduje z istniejącymi

GET /availability/my (wymaga JWT + doc role)
- Zwraca: Wszystkie sloty zalogowanego terapeuty
- Sortowanie: Po startTime

GET /availability/doc/:docId
- Zwraca: Wolne sloty konkretnego terapeuty
- Logika: Zwraca tylko sloty z booked=false i startTime > now

DELETE /availability/:slotId (wymaga JWT + doc role)
- Zwraca: Status usunięcia
- Logika: Usuwa slot, tylko jeśli nie jest zarezerwowany (booked=false)
- Walidacja: Wymaga verificationStatus=approved

AVAILABILITY SERVICE - METODY:
-------------------------------
- create(docId, dto): Tworzy nowy slot (wymaga verificationStatus=approved)
- findForDoc(docId): Pobiera wszystkie sloty terapeuty
- findFreeSlotsForDoc(docId): Pobiera wolne sloty terapeuty (tylko approved)
- remove(docId, slotId): Usuwa slot (jeśli niezarezerwowany, wymaga approved)

6. MEETINGS MODULE (/meetings)
-------------------------------
Zarządzanie spotkaniami między pacjentami a terapeutami.

ENDPOINTY:
----------

POST /meetings (wymaga JWT + user role)
- Body: CreateMeetingDto (slotId, clientMessage?)
- Zwraca: Utworzone spotkanie
- Logika: 
  1. Sprawdza czy slot jest wolny
  1a. Wymaga zaakceptowanych zgód (consentTerms, consentAdult, consentHealthData)
  2. Tworzy spotkanie ze statusem 'pending'
  3. Oznacza slot jako booked=true
  4. Kopiuje czasy ze slotu do spotkania

GET /meetings/me (wymaga JWT + user role)
- Query: scope? (upcoming | past | all)
- Zwraca: Lista spotkań pacjenta
- Filtracja: 
  - upcoming: status in (pending, confirmed)
  - past: status in (completed, cancelled_by_user, cancelled_by_doc, cancelled_by_system)
  - all: wszystkie

GET /meetings/doc (wymaga JWT + doc role)
- Query: scope? (today | upcoming | past | cancelled), page?, limit?
- Zwraca: { items, page, limit, total }
- items[]: { id, startTime, endTime, status, paid, clientName, clientMessage }
- Filtracja: 
  - today: dzisiejsze spotkania
  - upcoming: nadchodzące spotkania
  - past: przeszłe spotkania
  - cancelled: anulowane spotkania
- Paginacja: domyślnie page=1, limit=20

POST /meetings/:id/cancel (wymaga JWT + user role)
- Zwraca: Anulowane spotkanie
- Logika:
  1. Zmienia status na cancelled_by_user
  2. Zwalnia slot (booked=false)
  3. Jeśli spotkanie było confirmed, anulowanie tylko > 24h przed startem

POST /meetings/doc/:id/cancel (wymaga JWT + doc role)
- Zwraca: Anulowane spotkanie
- Logika:
  1. Zmienia status na cancelled_by_doc
  2. Zwalnia slot (booked=false)
  3. Terapeuta może anulować zawsze

MEETINGS SERVICE - METODY:
---------------------------
- createMeeting(userId, dto): Tworzy nowe spotkanie
- getUserMeetings(userId, scope): Pobiera spotkania pacjenta
- getDocMeetingsListForUser(docUserId, scope, page, limit): Pobiera spotkania terapeuty z paginacją
- cancelMeetingByUser(meetingId, userId): Anuluje spotkanie (pacjent)
- cancelMeetingByDocUser(meetingId, docUserId): Anuluje spotkanie (terapeuta)

MEETINGS CRON SERVICE:
----------------------
- @Cron('*/5 * * * *') - Co 5 minut
- markCompletedMeetings(): Automatycznie zmienia status spotkań które się zakończyły
  (endTime < now AND status = confirmed) na 'completed'
- autoCancelUnpaidMeetings(): Automatycznie anuluje spotkania pending starsze niż 15 minut
  (status=cancelled_by_system, zwalnia slot, P24 pending -> failed)

7. PAYMENTS MODULE (/payments)
-------------------------------
Obsługa płatności przez Stripe i Przelewy24.

ENDPOINTY:
----------

POST /payments/checkout (wymaga JWT + user role)
- Body: CreateCheckoutSessionDto (meetingId)
- Zwraca: { url, sessionId }
- Uwaga: Jeśli STRIPE_SECRET_KEY nie jest ustawione, Stripe jest wyłączony (503)
- Wymaga zaakceptowanych zgód (consentTerms, consentAdult, consentHealthData)
- Logika (Stripe):
  1. Weryfikuje spotkanie i cenę
  2. Tworzy Payment w bazie (status=pending, provider=stripe)
  3. Tworzy Stripe Checkout Session
  4. Zapisuje stripeCheckoutId w bazie
  5. Zwraca URL do płatności

POST /payments/p24/start (wymaga JWT + user role)
- Body: CreateCheckoutSessionDto (meetingId)
- Zwraca: { url, token, sessionId }
- Wymaga zaakceptowanych zgód (consentTerms, consentAdult, consentHealthData)
- Logika (Przelewy24):
  1. Weryfikuje spotkanie i cenę
  2. Tworzy Payment w bazie (status=pending, provider=p24)
  3. Generuje unikalny sessionId
  4. Rejestruje transakcję w P24 i zapisuje token
  5. Zwraca URL przekierowania

POST /payments/webhook (Stripe webhook)
- Body: Stripe event
- Header: stripe-signature
- Logika:
  1. Weryfikuje signature webhoka
  2. Obsługuje event 'checkout.session.completed':
     - Aktualizuje Payment (status=succeeded)
     - Aktualizuje Meeting (status=confirmed)
     - Zapisuje paymentIntentId

POST /payments/p24/webhook (P24 webhook)
- Body: P24 notification
- Logika:
  1. Weryfikuje dane od P24
  2. Aktualizuje Payment (status=succeeded/failed)
  3. Aktualizuje Meeting (status=confirmed)

PAYMENTS SERVICE - METODY:
---------------------------
- createCheckoutSessionForMeeting(userId, meetingId): Stripe checkout
- startP24PaymentForMeeting(userId, meetingId): Inicjalizacja P24
- handleStripeWebhook(rawBody, signature): Obsługa webhook Stripe
- handleP24Webhook(body): Obsługa webhook P24
- refundPaymentForMeeting(meetingId, reason): Zwrot przez Przelewy24
- p24SignRegister(params): Generuje podpis dla P24

8. EMAIL MODULE (/email)
-------------------------
Wysyłka emaili weryfikacyjnych i resetowania hasła przez Resend.

ENDPOINTY:
----------

POST /email/confirm
- Body: ConfirmEmailDto (token)
- Zwraca: Status potwierdzenia
- Logika:
  1. Dekoduje JWT token
  2. Ustawia emailConfirmed=true

POST /email/resend-confirm
- Body: EmailDto (email)
- Zwraca: Status wysyłki
- Logika:
  1. Sprawdza czy token nie był wysłany ostatnio (cooldown)
  2. Generuje JWT token z emailem
  3. Wysyła email z linkiem potwierdzającym
  4. Zapisuje tokenActivatedAt

POST /email/password
- Body: EmailDto (email)
- Zwraca: Status wysyłki
- Logika:
  1. Sprawdza cooldown
  2. Generuje token
  3. Wysyła email z linkiem do resetowania hasła

PATCH /email/password-change
- Body: ChangePasswordDto (token, newPassword)
- Zwraca: Status zmiany
- Logika:
  1. Weryfikuje token
  2. Hashuje nowe hasło
  3. Aktualizuje w bazie

EMAIL SERVICE - METODY:
------------------------
- confirmEmail(email): Potwierdza email
- decodeConfirmationToken(token): Dekoduje JWT token potwierdzenia
- resendConfirmationLink(email): Wysyła ponownie link potwierdzający
- sendPasswordLink(email): Wysyła link do resetu hasła
- changePassword(dto): Zmienia hasło na podstawie tokenu

9. ADMIN MODULE (/admin)
------------------------
Panel administratora do zarządzania weryfikacją terapeutów.

ENDPOINTY:
----------

GET /admin/documents/:id/url (wymaga JWT + admin role)
- Zwraca: { url } (signed URL do dokumentu weryfikacyjnego)

POST /admin/docs/:docProfileId/approve (wymaga JWT + admin role)
- Zwraca: Zaktualizowany profil terapeuty
- Logika: 
  1. Zmienia verificationStatus na 'approved'
  2. Zapisuje reviewedAt
  3. Terapeuta automatycznie staje się widoczny publicznie
  4. Może teraz dodawać sloty dostępności

POST /admin/docs/:docProfileId/reject (wymaga JWT + admin role)
- Body: { reason: string }
- Zwraca: Zaktualizowany profil terapeuty
- Logika:
  1. Zmienia verificationStatus na 'rejected'
  2. Zapisuje reviewedAt i rejectionReason
  3. Terapeuta musi poprawić profil i wysłać ponownie przez POST /profiles/doc/submit
  4. Terapeuta traci możliwość zarządzania slotami (jeśli miał)

ADMIN SERVICE - METODY:
------------------------
- approveDocProfile(docProfileId): Zatwierdza profil terapeuty (approved = widoczny publicznie)
- rejectDocProfile(docProfileId, reason): Odrzuca profil terapeuty z powodem (musi poprawić)

FLOW REJESTRACJI I LOGOWANIA:
------------------------------

1. LEKKA REJESTRACJA (pacjent):
   POST /auth/register-light { email, consentTerms, consentAdult, consentHealthData }
   -> Tworzy User (emailConfirmed=false) + UserProfile
   -> Zapisuje zgody (consentTerms, consentAdult, consentHealthData)
   -> Loguje użytkownika (cookies: access + refresh + flags)
   -> Wysyła email: ustaw hasło / potwierdź email
   -> Użytkownik może od razu: wyszukiwać terapeutów, bookować, płacić

2. POTWIERDZENIE EMAIL:
   POST /email/resend-confirm { email }
   -> Wysyła email z tokenem
   POST /email/confirm { token }
   -> Ustawia emailConfirmed=true

3. USTAWIENIE HASŁA:
   POST /auth/set-password { password } (z JWT)
   -> Hashuje i zapisuje hasło
   -> Ustawia emailConfirmed=true

4. LOGOWANIE:
   POST /auth/login { email, password }
   -> Weryfikuje dane
   -> Zwraca access_token + refresh_token w cookies

5. ODŚWIEŻENIE SESJI:
   POST /auth/refresh
   -> Rotuje refresh token i ustawia nowe cookies

FLOW PACJENTA (USER):
---------------------

1. Rejestracja + uzupełnienie profilu:
   POST /auth/register-light
   PUT /profiles/user { firstName, lastName, problems }

2. Wyszukiwanie terapeutów (ultra lekkie):
   GET /matches/problems -> lista problemów
   POST /matches/search-with-slots { problemId, from, to }
   -> zwraca listę terapeutów wraz z wolnymi slotami w danym oknie czasu
   -> user wybiera terapeute i konkretny slot z listy

3. Sprawdzanie dostępności:
   GET /availability/doc/:docId -> wolne sloty

4. Rezerwacja spotkania:
   POST /meetings { slotId, clientMessage? }
   -> Tworzy Meeting (status=pending)
   -> Rezerwuje slot (booked=true)

5. Płatność:
   POST /payments/checkout { meetingId }
   -> Przekierowanie do Stripe
   -> Po płatności webhook -> Meeting (status=confirmed)

   LUB

   POST /payments/p24/start { meetingId }
   -> Przekierowanie do Przelewy24
   -> Po płatności webhook -> Meeting (status=confirmed)

   Uwaga: potwierdzenie spotkania wynika z płatności, niezależnie od statusu potwierdzenia emaila.

6. Zarządzanie spotkaniami:
   GET /meetings/me?scope=upcoming -> nadchodzące
   GET /meetings/me?scope=past -> przeszłe
   POST /meetings/:id/cancel -> anulowanie (confirmed tylko >24h przed startem)

7. Ocena terapeuty (po spotkaniu):
   POST /profiles/rate { docId, meetingId, rate, comment? }

8. Sprawdzanie ocen terapeuty:
   GET /profiles/doc/:docId/ratings?page=1&limit=10

FLOW TERAPEUTY (DOC):
----------------------

1. Rejestracja jako terapeuta (admin musi ustawić role):
   POST /users { email, password, role: "doc" }

2. Uzupełnienie profilu:
   PUT /profiles/doc { 
     firstName, lastName, profession,
     sessionPricePln, specializations[], docTermsAccepted
   }

3. Dodanie dokumentów weryfikacyjnych (wymagane):
   POST /doc/verification-documents (multipart)
   GET /doc/verification-documents

4. Przesłanie profilu do weryfikacji:
   POST /profiles/doc/submit
   -> verificationStatus = 'submitted'

5. Oczekiwanie na weryfikację admina:
   - Admin zatwierdza: POST /admin/docs/:id/approve
     → verificationStatus = 'approved' (profil automatycznie widoczny publicznie)
   - Admin odrzuca: POST /admin/docs/:id/reject { reason }
     → verificationStatus = 'rejected' (terapeuta musi poprawić i wysłać ponownie)

6. Po zatwierdzeniu - zarządzanie dostępnością:
   UWAGA: Tylko zatwierdzeni terapeuci (approved) mogą dodawać/usuwać sloty
   POST /availability { startTime, endTime }
   GET /availability/my -> wszystkie sloty
   DELETE /availability/:slotId -> usuwanie (jeśli niezarezerwowane)

7. Zarządzanie spotkaniami:
   GET /meetings/doc?scope=today&page=1&limit=20
   GET /meetings/doc?scope=upcoming
   GET /meetings/doc?scope=past
   POST /meetings/doc/:id/cancel -> anulowanie (doc zawsze)

8. Kalendarz tygodniowy:
   GET /doc/calendar/week?from=2024-01-01&to=2024-01-07

9. Monitorowanie opinii:
   GET /profiles/doc -> sprawdzanie swojej oceny (rate, ratesLot)
   GET /profiles/doc/:docId/ratings -> lista opinii

FLOW PŁATNOŚCI:
---------------

STRIPE:
1. Użytkownik rezerwuje spotkanie (status=pending)
2. POST /payments/checkout -> tworzy Payment + Stripe session
3. Użytkownik płaci w Stripe
4. Stripe wysyła webhook -> /payments/webhook
5. Backend:
   - Weryfikuje webhook
   - Aktualizuje Payment (status=succeeded)
   - Aktualizuje Meeting (status=confirmed)

PRZELEWY24:
1. Użytkownik rezerwuje spotkanie (status=pending)
2. POST /payments/p24/start -> tworzy Payment + dane do P24
3. Frontend przekierowuje do P24 z sessionId
4. Użytkownik płaci w P24
5. P24 wysyła webhook -> /payments/p24/webhook
6. Backend:
   - Weryfikuje dane P24
   - Aktualizuje Payment (status=succeeded)
   - Aktualizuje Meeting (status=confirmed)

FLOW ANULOWANIA:
----------------
1. Użytkownik/terapeuta anuluje spotkanie:
   POST /meetings/:id/cancel lub POST /meetings/doc/:id/cancel
2. Backend:
   - Zmienia status Meeting (cancelled_by_user/cancelled_by_doc)
   - Zwalnia slot (booked=false)
   - Jeśli Payment.status=succeeded i provider=p24 -> zwrot przez Przelewy24

AUTOMATYZACJE (CRON JOBS)
===========================================

1. Meetings Cron (co 5 minut):
   - Znajduje spotkania z endTime < now i status=confirmed
   - Zmienia status na 'completed'

2. Auto-anulowanie nieopłaconych spotkań (co 5 minut):
   - Znajduje spotkania z status=pending starsze niż 15 minut
   - Zmienia status na 'cancelled_by_system'
   - Zwalnia slot (booked=false)
   - Aktualizuje powiązane płatności (P24 pending -> failed)

3. Cleanup Cron (obecnie wyłączony):
   - Miałby usuwać użytkowników z niepotwierdzonym emailem

===========================================
ZABEZPIECZENIA
===========================================

GUARDS:
-------
1. JwtAuthGuard: Weryfikuje JWT token z cookie
2. RolesGuard: Sprawdza rolę użytkownika (@Roles decorator)

AUTENTYKACJA:
-------------
- Access token w httpOnly cookie (bezpieczniejsze niż localStorage)
- Refresh token w httpOnly cookie (path=/auth/refresh)
- Token zawiera userId
- Ważność: access domyślnie 15 minut, refresh domyślnie 7 dni
- Rotacja refresh tokenów + limit aktywnych sesji (MAX_SESSIONS_PER_USER)

AUTORYZACJA:
------------
- Dekoratory @Roles([Role.user]) określają wymagane role
- Guards sprawdzają czy użytkownik ma odpowiednią rolę
- Użytkownik może modyfikować tylko swoje dane

HASŁA:
------
- Hashowanie bcrypt (10 rounds)
- Walidacja siły hasła w DTO

WEBHOOKS:
---------
- Stripe: Weryfikacja podpisu webhoka
- P24: Weryfikacja CRC/signature

WALIDACJA:
----------
- ValidationPipe (class-validator) na wszystkich endpointach
- DTO z dekoratora mi walidacyjnymi
- Whitelist: true (usuwa nieznane pola)

THROTTLING:
-----------
- Domyślnie: 60 requestów na 60 sekund
- Login: 5 requestów na 15 minut (900 sek)
- Register-light: 5 requestów na 15 minut
- Set-password: 3 requesty na 15 minut
- Logout: bez limitu (SkipThrottle)

WERYFIKACJA TERAPEUTÓW:
-----------------------
- Tylko admin może zatwierdzać/odrzucać terapeutów
- verificationStatus kontroluje widoczność publiczną:
  * draft - profil w edycji, niewidoczny
  * submitted - czeka na weryfikację admina, niewidoczny
  * approved - zatwierdzony, WIDOCZNY PUBLICZNIE, może zarządzać slotami
  * rejected - odrzucony, niewidoczny, wymaga poprawek
- Tylko terapeuci z approved mogą:
  * Dodawać/usuwać sloty dostępności
  * Być widoczni w wynikach wyszukiwania
  * Być dopasowywani do pacjentów
  * Otrzymywać rezerwacje

CORS:
-----
- Origin: process.env.CLIENT_URL
- Credentials: true (cookies)

===========================================
PRZYKŁADOWE FLOW KODU
===========================================

PRZYKŁAD: Tworzenie spotkania
------------------------------

1. Request: POST /meetings
   Body: { slotId: 42, clientMessage: "Witam" }

2. MeetingsController.createMeeting()
   - Weryfikuje JWT (JwtAuthGuard)
   - Sprawdza rolę (RolesGuard - Role.user)
   - Wywołuje MeetingsService.createMeeting()

3. MeetingsService.createMeeting()
   - Pobiera slot z bazy
   - Sprawdza czy slot jest wolny (booked=false)
   - Tworzy Meeting w transakcji (atomiczne zarezerwowanie slotu):
     * Tworzy rekord Meeting
     * Aktualizuje slot (booked=true)
   - Zwraca utworzone spotkanie

4. Response: MeetingEntity (serializowany)

PRZYKŁAD: Webhook Stripe
-------------------------

1. Request: POST /payments/webhook
   Header: stripe-signature: xxx
   Body: Stripe event JSON (raw)

2. PaymentsController.handleStripeWebhook()
   - Pobiera raw body i signature
   - Wywołuje PaymentsService.handleStripeWebhook()

3. PaymentsService.handleStripeWebhook()
   - Weryfikuje signature (stripe.webhooks.constructEvent)
   - Jeśli event.type === 'checkout.session.completed':
     * Pobiera paymentId i meetingId z metadata
     * W transakcji:
       - Aktualizuje Payment (status=succeeded)
       - Aktualizuje Meeting (status=confirmed)
     * Loguje sukces

4. Response: { received: true }
